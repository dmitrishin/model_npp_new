## Полный список критических замечаний

### Участник 1 (15 критик)

**S03-C-1-01** → S03-R1-4-01
Трудоёмкость начального развёртывания «цифрового двойника» не оценена — может сама составлять месяцы работы; обещание экономии 6–12 месяцев декларативно.

**S03-C-1-02** → S03-R1-4-02
Обещание сокращения сроков экспертизы на 30–50% не подкреплено. Семантический слой проверяет терминологию, но не содержание обоснования безопасности; основные задержки лицензирования — содержательные вопросы регулятора.

**S03-C-1-03** → S03-R1-5-05
Инвариант ацикличности — стандартная топологическая сортировка, решаемая в любом планировщике. Основные сложности Open Top — пространственно-геометрические ограничения (краны, зоны подъёма), которые модель не формализует.

**S03-C-1-04** → S03-R1-6-12
Автогенерация графиков 3–4 уровней переносит проблему из «ручного составления» в «ручное создание шаблонов». Графики 4 уровня зависят от конкретных фронтов, ресурсов и состояния площадки.

**S03-C-1-05** → S03-R1-5-12
ML-генерация ИТТ — юридически значимый документ; трудоёмкость проверки ML-результата может быть сопоставима с ручной разработкой, а ответственность несёт инженер.

**S03-C-1-06** → S03-R1-2-09
Социально-бытовая инфраструктура — за пределами предметной области модели (c_phys и c_func для столовой не имеют того же значения, что для парогенератора). Системный характер модели размывается.

**S03-C-1-07** → S03-R1-6-05
GERT-подграфы дают вероятностные распределения сроков, но перевод в финансовые распределения требует отдельной финансовой модели, не описанной в статье.

**S03-C-1-08** → S03-R1-4-06
Обещание сокращения основного периода на 2–4 месяца без обоснования механизма. ТМР — физическая работа, модель не может ускорить сварку.

**S03-C-1-09** → S03-R1-3-14
«Единая управленческая панель» — фактически BI-дашборд, а не применение формальной модели. Требует отдельного слоя визуализации, UX, прав доступа.

**S03-C-1-10** → S03-R1-6-06
Разделение c_phys/c_func — один элемент модели, стандартная практика в ПНР. Не обосновано, зачем для этого нужна полная модель M-S-D-Q-U.

**S03-C-1-11** → S03-R1-6-14
Предиктивная аналитика на матрицах графа требует объёмов обучающих данных, недоступных в ядерном строительстве (единицы проектов).

**S03-C-1-12** → S03-R1-3-02
Прогнозирование «вероятности возврата» регулятором — для каждой страны единицы кейсов, статистика недостоверна. Ключевые задержки — действия регулятора, не зависящие от модели.

**S03-C-1-13** → S03-R1-2-03
Формализация передачи помещений через полную модель M-S-D-Q-U несоразмерна масштабу оперативной задачи; система может отставать от реального темпа стройки.

**S03-C-1-14** → S03-R1-4-10
Кросс-проектное ML требует унификации данных из разных ИТ-систем и классификаторов; есть ограничения конфиденциальности экспортных проектов.

**S03-C-1-15** → S03-R1-1-10
Оптимизация СМБ — задача MRP/MES, для которой полная модель M-S-D-Q-U — несоразмерное усложнение.

### Участник 2 (20 критик)

**S03-C-2-01** → S03-R1-4-01
Слишком широкая идея — многолетняя программа цифровой трансформации. Не выделен первый контур, дающий быстрый эффект.

**S03-C-2-02** → S03-R1-4-02
Переоценена автоматизируемость лицензирования. Решение регулятора зависит от качества обоснований, очных замечаний и национальной практики.

**S03-C-2-03** → S03-R1-3-02
Переобучение по данным прошлых проектов — выборка сопоставимых кейсов мала, оценки статистически неустойчивы.

**S03-C-2-04** → S03-R1-5-01
Жёсткий порог 85% маскирует критичность отдельных позиций — один недостающий комплект по котловану блокирует больше, чем десятки второстепенных документов.

**S03-C-2-05** → S03-R1-1-04
Узкое место потока РД — поздние ИДП от изготовителей и повторные изменения после контрактации. Без управления качеством внешних ИДП модель описывает симптомы, а не причину.

**S03-C-2-06** → S03-R1-5-03
Буквальный перенос just-in-time на ядерный проект с длинной логистикой, таможней, санкционными рисками. Для критичных позиций нужны буферы.

**S03-C-2-07** → S03-R1-6-02
Привязка поставок к T_need может быть запоздалой для ОДЦИ, где закупочное окно определяется на годы раньше.

**S03-C-2-08** → S03-R1-5-05
Ацикличность доказывает логическую упорядоченность, но не технологическую реализуемость крупноблочного монтажа (краны, опоры, heavy-lift engineering).

**S03-C-2-09** → S03-R1-6-04
Атрибут Local/Foreign слишком груб — важны доля местной добавленной стоимости, nuclear QA, границы перепроектирования.

**S03-C-2-10** → S03-R1-5-07
Максимизация локализации может конфликтовать со сроками, лицензированием и качеством. Нет критерия остановки.

**S03-C-2-11** → S03-R1-3-14
Заказчик, генподрядчик и генпроектировщик не согласятся на единые статусы и метрики без контура data governance.

**S03-C-2-12** → S03-R1-4-10
Перенос моделей между проектами неочевиден из-за различий в странах, контрактах, подрядчиках. Без нормализации — риск ложных закономерностей.

**S03-C-2-13** → S03-R1-5-12
Для safety-related оборудования ИТТ требуют инженерного замысла; ML может тиражировать ошибки аналога.

**S03-C-2-14** → S03-R1-6-12
Префиксный код и шаблон не содержат информации о методе производства, доступах, сезонности, стратегии подрядчика.

**S03-C-2-15** → S03-R1-1-06
Видимость точной вероятностной оценки при недостаточной статистике по редким событиям. Monte Carlo на субъективных допущениях.

**S03-C-2-16** → S03-R1-1-07
Полная трассируемость десятков тысяч требований не решает конфликт между ними и не показывает критичность. Риск дорогой в поддержке системы.

**S03-C-2-17** → S03-R1-1-08
min τ_func слишком «хрупкая» метрика — любой мелкий хвост равен полноценному блокеру. Нет приоритизации стоп-факторов.

**S03-C-2-18** → S03-R1-6-14
Матрицы смежности/лапласиана слабо ловят причинность задержек (квалификация подрядчика, погода, регуляторные замечания).

**S03-C-2-19** → S03-R1-3-06
График «от ввода назад» на ранних этапах может создавать ложную детерминированность при незафиксированных конфигурации и стратегии.

**S03-C-2-20** → S03-R1-4-04
Агрегирование рисков по иерархии маскирует единичные low-frequency/high-impact события (провал лицензирования, срыв EPC).

### Участник 3 (12 критик)

**S03-C-3-01** → S03-R1-4-01
Слишком широкая идея; неясно, какое управленческое решение улучшится первым и где быстрый эффект.

**S03-C-3-02** → S03-R1-4-02
Основной риск лицензирования — внешний цикл рассмотрения регулятором; слабо показано влияние модели на эти внешние ограничения.

**S03-C-3-03** → S03-R1-2-10
PIM/K-PIM-платформа: бизнес-ценность косвенная и отложенная, прямой механизм влияния на критические события сооружения не показан.

**S03-C-3-04** → S03-R1-4-10
Межпроектное ML-хранилище стратегически полезно, но для конкретного проекта ценность удалена от текущих узких мест.

**S03-C-3-05** → S03-R1-5-12
Автогенерация ИТТ рискует перенести устаревшие допущения и размывает персональную инженерную ответственность.

**S03-C-3-06** → S03-R1-1-06
При низком качестве исходных экспертных данных точность вероятностной модели иллюзорна, а трудоёмкость сопровождения чрезмерна.

**S03-C-3-07** → S03-R1-1-03
Недостаточно учтены локализация, таможня, инспекции, условия EPC и готовность площадки к входному контролю (экспортные проекты).

**S03-C-3-08** → S03-R1-2-03
Фокус на помещениях может привести к локальной оптимизации; системные межобъектные зависимости раскрыты слабо.

**S03-C-3-09** → S03-R1-2-08
Не показан критерий границы, после которой вынос работ в СМБ ухудшает ситуацию из-за транспортных ограничений.

**S03-C-3-10** → S03-R1-5-07
Формальные замены могут конфликтовать с зрелостью поставщика, квалификацией производства и надёжностью оборудования.

**S03-C-3-11** → S03-R1-3-14
Панель останется отчётной витриной без решённых конфликтов источников данных и арбитража между сторонами.

**S03-C-3-12** → S03-R1-3-10
Сценарная оптимизация инвестиций чувствительна к предположениям; без статистической базы — предмет спора, а не основа решения.

### Участник 4 (8 критик)

**S03-C-4-01** → S03-R1-1-02
Новые требования регулятора часто требуют нарушения инвариантов модели. Модель не учитывает внешнюю сертификацию AI-инструмента для ядерной безопасности.

**S03-C-4-02** → S03-R1-4-01
Полная интеграция требует единого источника данных из несвязанных систем (Primavera, 1C, АСУ ТП, СЭД); миграция — многолетняя и дорогая.

**S03-C-4-03** → S03-R1-1-03
GERT не учитывает геополитические и санкционные риски; импортозамещение может требовать 18–24 месяца дополнительно.

**S03-C-4-04** → S03-R1-2-06
Автоматическая блокировка при Φ = ∅ игнорирует практику коллегиального утверждения изменений (ГРП, авторский надзор, регулятор).

**S03-C-4-05** → S03-R1-3-09
Для сквозной прослеживаемости 10 000+ требований значительная часть документации на бумаге/сканах и на разных языках. Дорогостоящая разметка.

**S03-C-4-06** → S03-R1-5-01
Жёсткий предикат Trans_k = 1 не предусматривает режим «управляемого допуска с отклонениями» (waivers), реально применяемый при первом бетоне.

**S03-C-4-07** → S03-R1-6-14
ML на матричных представлениях требует полных графов десятков проектов; кросс-проектное обучение невозможно из-за конфиденциальности.

**S03-C-4-08** → S03-R1-4-10
Прямой перенос конфигураций может привести к несоответствию лицензионным требованиям страны-заказчика.

### Участник 5 (8 критик)

**S03-C-5-01** → S03-R1-1-04
Горизонтальные гиперрёбра для зависимостей между комплектами РД противоречат инварианту ацикличности — проектирование смежных дисциплин итеративно и содержит циклические ссылки.

**S03-C-5-02** → S03-R1-2-13
Жёсткая привязка прогресса к Evidence(x) в D создаст рассинхронизацию с реальностью из-за бюрократического лага оформления документов (недели).

**S03-C-5-03** → S03-R1-3-06
Обратное планирование с высокостохастичными процессами (лицензирование, контрактация) приведёт к уходу расчётных дат позднего старта в прошлое.

**S03-C-5-04** → S03-R1-4-10
Топология графа строительства сильно меняется в зависимости от подрядчика и площадки; прямой перенос генерирует нереализуемые последовательности. Нарушение NDA зарубежных заказчиков.

**S03-C-5-05** → S03-R1-6-12
KKS — функционально-технологический код, а строительство ведётся по отметкам и захваткам. Автогенерация создаст топологию, оторванную от физического потока СМР.

**S03-C-5-06** → S03-R1-1-07
Отображения ψ и ω для высокоуровневых требований (сейсмостойкость) приведут к комбинаторному взрыву — Impact Analysis будет выдавать весь проект как затронутую зону.

**S03-C-5-07** → S03-R1-6-14
Матрица Лапласа описывает статичную связность, а задержки генерируются динамическими факторами. Обучение на структурных матрицах — ложные корреляции.

**S03-C-5-08** → S03-R1-2-04
Моделирование каждого стыка/опоры как вершины — комбинаторный взрыв (сотни тысяч вершин); вычисление матриц вычислительно неподъёмно для оперативного управления.

### Участник 6 (6 критик)

**S03-C-6-01** → S03-R1-1-01
Графики 4-го уровня — сотни изменений в день; поддержание инварианта целостности при такой частоте приведёт к постоянному пересчёту или блокировке.

**S03-C-6-02** → S03-R1-2-01
Жёсткая привязка перехода к формальным инвариантам может остановить работы из-за незначительного несоответствия. Нет механизма «управляемого допуска с отклонениями» (waivers).

**S03-C-6-03** → S03-R1-3-03
Трудоёмкость наполнения S (тысячи нормативных документов) — ресурсы сопоставимы с самим проектированием.

**S03-C-6-04** → S03-R1-4-10
Гетерогенность данных: выборка мала (единицы проектов), дисперсия условий велика — ложные корреляции.

**S03-C-6-05** → S03-R1-5-05
Граф M статичен в моменте; модель может разрешить монтаж, невозможный из-за ветра — погодные факторы в жёсткой топологии не учтены.

**S03-C-6-06** → S03-R1-5-12
Риск «галлюцинаций» ML при генерации юридически значимых ИТТ; перенос скрытых ошибок без участия эксперта.
